Слайд 1 — Вступ: що таке API та інтеграції
Скрипт викладача:
“Сьогодні ми поговоримо про одну з найважливіших тем сучасної розробки — API та інтеграції.
Практично кожен сучасний застосунок не існує окремо. Він постійно взаємодіє з іншими сервісами, базами даних, платіжними системами, мобільними додатками чи навіть штучним інтелектом.
Почнемо з поняття API.
API — це Application Programming Interface, тобто інтерфейс програмування застосунків.
Якщо простими словами — це набір правил, який дозволяє різним програмам спілкуватися між собою.
Можна уявити API як офіціанта в ресторані 🍽️.
Клієнт не йде напряму на кухню готувати собі їжу. Він робить замовлення через офіціанта.
Офіціант передає запит на кухню, а потім приносить результат назад.
Так само і API:
клієнтський застосунок робить запит,
сервер його обробляє,
і повертає відповідь.
Наприклад:
мобільний банкінг отримує ваш баланс через API,
Instagram завантажує стрічку через API,
сайт доставки отримує геолокацію через Google Maps API.
Далі — інтеграції.
Інтеграція — це процес об’єднання різних систем або сервісів для спільної роботи.
Наприклад:
сайт інтернет-магазину інтегрується з платіжною системою,
CRM інтегрується з Telegram або email-сервісом,
застосунок може інтегрувати ChatGPT для роботи AI-функцій.
Сьогодні майже весь сучасний софт побудований на інтеграціях.
Компанії рідко створюють усе з нуля — вони підключають готові сервіси через API.
Чому це важливо?
Тому що API та інтеграції дають багато переваг:
повторне використання готової логіки,
швидшу розробку,
простіше масштабування,
можливість будувати великі розподілені системи.
Наприклад, замість створення власної карти світу — розробники просто використовують Google Maps API. Це економить роки роботи.
На цій лекції ми розглянемо:
як працюють API,
що таке REST і GraphQL,
як відбувається інтеграція сервісів,
як читати API документацію,
та як розробники тестують API на практиці.
Тема дуже практична, тому по ходу лекції будемо розбирати реальні приклади із сучасних продуктів.”

Слайд 2 — Компоненти API
Скрипт викладача:
“Тепер давайте розберемося, з чого взагалі складається API.
Коли ми говоримо про API, важливо розуміти, що це не просто одна URL-адреса. Це ціла система взаємодії між програмами.
Перший важливий компонент — це ендпоінти або endpoints.
Ендпоінт — це конкретна адреса API, куди ми можемо відправити запит.
Наприклад:
/users
/products
/orders
Кожен ендпоінт виконує певну задачу.
Один може повертати список користувачів, інший — створювати замовлення.
Тут важливе поняття контракту API.
Контракт — це домовленість між клієнтом і сервером:
який запит потрібно відправити,
які параметри передати,
і яку відповідь ми отримаємо.
Якщо фронтенд очікує одне, а бекенд повертає інше — система починає ламатися.
Тому API-контракт — це дуже важлива частина розробки.
Далі — формати повідомлень.
Програми мають якось передавати дані одна одній. Найчастіше сьогодні використовується формат JSON.
Наприклад:
{
"name": "Vlad",
"age": 25
}
JSON популярний тому, що:
він простий,
читабельний,
і підтримується майже всіма мовами програмування.
Також існують XML, Protocol Buffers та інші формати, але JSON зараз — стандарт для більшості веб API.
Окрім формату, існують ще схеми даних.
Схема описує:
які поля обов’язкові,
який тип даних має поле,
чи можна передати null,
які є обмеження.
Наприклад:
age має бути числом,
email — рядком,
id — унікальним значенням.
Далі — клієнт і сервер.
Клієнт — це той, хто робить запит:
браузер,
мобільний застосунок,
frontend,
або навіть інший сервер.
Сервер — це система, яка приймає запит, обробляє його та повертає відповідь.
Наприклад:
React frontend відправляє запит,
Spring Boot backend обробляє його,
база даних повертає інформацію,
а сервер надсилає результат назад у frontend.
І останній важливий компонент — документація API.
API без документації майже неможливо нормально використовувати 😄
Документація описує:
доступні ендпоінти,
параметри,
приклади запитів,
формати відповідей,
коди помилок,
авторизацію.
Найпопулярніший інструмент сьогодні — Swagger або OpenAPI.
Також часто є тестові середовища:
sandbox,
staging,
test environment.
Вони дозволяють безпечно тестувати інтеграції без впливу на реальних користувачів або реальні гроші.
Наприклад, платіжні системи мають тестові API, де можна “оплачувати” товари тестовими картками.
У реальних проєктах розуміння всіх цих компонентів дуже важливе, тому що API — це основа взаємодії сучасних систем.”

Слайд 3 — Типи API за доступом
Скрипт викладача:
“API можуть відрізнятися не тільки за технологією, а й за тим, хто має до них доступ.
Тому зараз розглянемо основні типи API за рівнем доступу.
Перший тип — це публічні або Public API.
Такі API відкриті для зовнішніх розробників.
Будь-хто може зареєструватися, отримати ключ доступу та використовувати їх у своїх застосунках.
Приклади:
Google Maps API,
OpenWeather API,
Stripe API,
GitHub API.
Саме завдяки публічним API тисячі сервісів можуть інтегрувати готові можливості:
карти,
платежі,
авторизацію,
AI,
аналітику.
Наприклад, коли сайт показує карту доставки — дуже часто це Google Maps API.
Або коли ви логінитесь через Google чи Facebook — це також інтеграція через API.
Другий тип — партнерські або Partner API.
Такі API доступні лише певним компаніям або партнерам.
Наприклад:
служба доставки може надати API тільки великим інтернет-магазинам,
банк може відкрити API лише перевіреним партнерам,
авіакомпанії дають API сервісам бронювання квитків.
Тут уже є:
договори,
обмеження,
спеціальні ключі доступу,
іноді VPN або IP whitelist.
Третій тип — внутрішні або Internal API.
Це API, які використовуються всередині компанії.
Наприклад:
frontend спілкується з backend,
один мікросервіс звертається до іншого,
CRM система взаємодіє з внутрішнім сервісом аналітики.
Такі API не публікуються назовні і часто доступні лише у внутрішній мережі компанії.
У сучасних мікросервісних архітектурах внутрішніх API може бути сотні або навіть тисячі.
І останній тип — приватні API.
Тут важливо не плутати з internal API.
Приватний API може існувати навіть у публічному продукті, але доступ до нього суворо обмежений.
Наприклад:
мобільний додаток може використовувати API, яке офіційно недоступне стороннім розробникам,
деякі функції Instagram або TikTok працюють через приватні API.
Компанії часто обмежують доступ через:
токени,
авторизацію,
rate limiting,
перевірку пристроїв,
шифрування.
Чому це важливо?
Тому що API — це точка входу в систему.
Якщо зробити її відкритою без контролю — можна отримати:
витік даних,
перевантаження сервера,
або проблеми з безпекою.
Тому рівень доступу до API — це дуже важлива частина архітектури сучасних систем.”

Слайд 4 — Архітектури API
Скрипт викладача:
“Тепер поговоримо про різні архітектури API.
Коли системи взаємодіють між собою, вони мають домовитися не лише про формат даних, а й про правила комунікації.
Існує кілька основних підходів, які сьогодні використовуються у розробці.
Почнемо з REST.
REST — це найпопулярніша архітектура API у веб-розробці.
REST означає Representational State Transfer.
Не треба запам’ятовувати складне визначення 😄
Головна ідея REST — використовувати звичайні HTTP-запити для роботи з даними.
REST API працює через:
URL,
HTTP методи,
JSON-відповіді.
Основні HTTP методи:
GET — отримати дані,
POST — створити,
PUT або PATCH — оновити,
DELETE — видалити.
Наприклад:
GET /users — отримати користувачів,
POST /orders — створити замовлення.
Чому REST став таким популярним?
Тому що він:
простий,
зрозумілий,
добре працює через інтернет,
і підтримується практично всіма технологіями.
Саме REST використовується у більшості сучасних веб і мобільних застосунків.
Далі — SOAP.
SOAP — це старіший і більш “формальний” стандарт API.
Він активно використовувався у великих enterprise-системах:
банках,
державних сервісах,
корпоративних інтеграціях.
SOAP працює через XML і має дуже строгі правила.
Його переваги:
високий рівень стандартизації,
вбудована безпека,
підтримка складних транзакцій.
Але SOAP часто складніший у використанні:
XML великий і менш читабельний,
запити більш громіздкі,
розробка займає більше часу.
Тому сьогодні SOAP використовується рідше, але в enterprise-середовищі він досі зустрічається.
Третій підхід — GraphQL.
GraphQL створила компанія Meta для вирішення проблем REST.
У REST часто виникає проблема:
або сервер повертає забагато даних,
або навпаки — недостатньо.
GraphQL дозволяє клієнту самому вказати, які саме поля потрібні.
Наприклад:
клієнт може попросити:
тільки ім’я користувача,
email,
і список замовлень.
І сервер поверне лише ці дані.
Це дуже зручно для:
мобільних застосунків,
складних frontend систем,
великих SPA застосунків на React або Angular.
Але GraphQL складніший у реалізації та кешуванні, ніж REST.
І останній підхід — gRPC.
gRPC створила Google.
Його головна ціль — дуже швидка взаємодія між сервісами.
На відміну від REST:
gRPC використовує не JSON,
а компактний бінарний формат Protocol Buffers.
Через це:
запити менші,
швидші,
і займають менше мережевого трафіку.
gRPC дуже популярний у:
мікросервісах,
high-load системах,
backend-to-backend комунікації.
Особливо коли сервіси написані різними мовами:
Java,
Go,
Kotlin,
Python,
C#.
Тому що gRPC добре підтримує міжмовну взаємодію.
Якщо коротко:
REST — простий і найпоширеніший,
SOAP — строгий enterprise-стандарт,
GraphQL — гнучке отримання даних,
gRPC — швидкість і ефективність для мікросервісів.
У реальних компаніях часто використовуються одразу кілька підходів залежно від задач.”

Слайд 5 — Протоколи та формати даних
Скрипт викладача:
“Щоб API могли працювати, системи повинні домовитися про дві речі:
як передавати дані,
і в якому форматі ці дані будуть зберігатися всередині повідомлень.
Тому зараз поговоримо про протоколи та формати даних.
Почнемо з HTTP та HTTPS.
HTTP — це основний протокол вебу.
Саме через нього браузер відкриває сайти, а frontend спілкується з backend.
Коли frontend надсилає API-запит — він фактично відправляє HTTP request.
Наприклад:
GET запит на отримання списку товарів,
POST запит на створення користувача.
HTTPS — це захищена версія HTTP.
Літера S означає Secure.
HTTPS використовує шифрування SSL/TLS.
Це означає, що:
паролі,
токени,
банківські дані
не передаються відкритим текстом через мережу.
Сьогодні майже всі сучасні API працюють саме через HTTPS.
Далі — WebSocket.
HTTP добре працює для класичних запитів:
клієнт запросив,
сервер відповів.
Але іноді потрібен постійний зв’язок у реальному часі.
Наприклад:
чат,
онлайн-гра,
біржа криптовалют,
live notifications.
Тут використовується WebSocket.
WebSocket створює постійне двостороннє з’єднання між клієнтом і сервером.
Тобто сервер може сам “штовхати” дані клієнту без нового запиту.
Саме тому повідомлення в Telegram або Discord приходять миттєво.
Тепер перейдемо до форматів даних.
Найпопулярніший формат сьогодні — JSON.
JSON:
легкий,
читабельний,
простий для людей і програм.
Приклад:
{
"name": "Anna",
"role": "QA"
}
JSON став стандартом для REST API.
Далі — XML.
XML старіший і більш формальний.
Він виглядає приблизно так:
<user>
<name>Anna</name>
<role>QA</role>
</user>
XML часто використовується у:
SOAP,
enterprise системах,
державних сервісах,
банківських інтеграціях.
Його плюс — строгість і можливість складної валідації.
Але XML:
більший за розміром,
менш зручний,
і складніше читається.
І ще один формат — Protocol Buffers або Protobuf.
Його активно використовує gRPC.
На відміну від JSON та XML, Protobuf — це бінарний формат.
Людина його майже не читає напряму, але він:
дуже компактний,
швидкий,
ефективний.
Тому він чудово підходить для:
high-load систем,
мікросервісів,
систем з великою кількістю запитів.
Тепер важливе поняття — Content-Type.
Коли клієнт надсилає запит, він має повідомити серверу, у якому форматі знаходяться дані.
Для цього використовується заголовок Content-Type.
Наприклад:
application/json
application/xml
application/protobuf
Сервер дивиться на цей заголовок і розуміє, як правильно обробити інформацію.
І останнє поняття — серіалізація.
Серіалізація — це процес перетворення об’єктів програми у формат для передачі через мережу.
Наприклад:
Java-об’єкт або Kotlin data class перетворюється у JSON.
А коли сервер отримує JSON назад і перетворює його у внутрішній об’єкт — це називається десеріалізація.
Це одна з базових речей, які постійно відбуваються “під капотом” будь-якого API.”
Слайд 6 — RESTful принципи
Скрипт викладача:
“Ми вже згадували REST як найпопулярнішу архітектуру API.
Тепер давайте детальніше розберемо основні RESTful принципи.
Перший принцип — ресурси та URI.
У REST усе представляється як ресурс.
Ресурсом може бути:
користувач,
товар,
замовлення,
коментар,
файл.
Кожен ресурс має свою унікальну адресу — URI або URL.
Наприклад:
/users
/products/15
/orders/120
Тобто URL має описувати саме ресурс, а не дію.
REST рекомендує уникати назв типу:
/getUsers
/deleteProduct
Замість цього використовуються HTTP методи.
І це другий важливий принцип.
HTTP методи описують, що саме ми хочемо зробити з ресурсом.
Основні методи:
GET — отримати дані,
POST — створити новий ресурс,
PUT або PATCH — оновити,
DELETE — видалити.
Наприклад:
GET /users/5 — отримати користувача,
POST /users — створити нового,
DELETE /users/5 — видалити користувача.
Тобто URL описує ресурс, а HTTP метод — дію.
Це робить API:
зрозумілим,
передбачуваним,
стандартизованим.
Далі — stateless або “без стану”.
REST API має бути stateless.
Це означає, що сервер не зберігає контекст між запитами.
Кожен запит повинен містити всю необхідну інформацію:
токен авторизації,
параметри,
дані користувача.
Сервер не має “пам’ятати”, що було в попередньому запиті.
Чому це важливо?
Тому що це:
спрощує масштабування,
дозволяє балансувати навантаження,
і робить систему більш стабільною.
Наприклад:
будь-який сервер у кластері може обробити запит незалежно від попередніх.
Ще один важливий принцип — кешування.
REST дозволяє кешувати відповіді сервера.
Наприклад:
список товарів,
картинки,
публічні дані.
Якщо дані не змінювалися — клієнт може не робити повторний запит.
Це:
пришвидшує роботу застосунку,
зменшує навантаження на сервер,
економить трафік.
Тому браузери, CDN та API Gateway активно використовують кешування.
І останній принцип — уніформний інтерфейс.
Це означає, що API має бути однаковим та передбачуваним у всій системі.
Наприклад:
однакові правила URL,
однакові формати помилок,
однакові HTTP статуси,
однаковий стиль назв.
Наприклад:
200 OK — успіх,
404 Not Found — ресурс не знайдено,
500 Internal Server Error — помилка сервера.
Коли API побудований консистентно — його легше:
підтримувати,
тестувати,
документувати,
і використовувати іншим командам.
Саме тому REST став стандартом у сучасній веб-розробці.”

Слайд 7 — Аутентифікація й авторизація
Скрипт викладача:
“Коли ми відкриваємо API назовні, одразу виникає питання безпеки.
Система повинна розуміти:
хто робить запит,
і що саме цьому користувачу дозволено.
Тут є два важливих поняття:
аутентифікація,
авторизація.
Аутентифікація відповідає на питання:
‘Хто ти?’
Авторизація:
‘Що тобі дозволено робити?’
Наприклад:
ви ввели логін і пароль — це аутентифікація.
А право редагувати користувачів або видаляти замовлення — це вже авторизація.
Почнемо з найпростішого способу — API keys.
API key — це спеціальний ключ доступу.
Клієнт додає його у запит, і сервер перевіряє:
чи ключ валідний,
чи має доступ до API.
Наприклад:
Authorization: ApiKey xxxxxx
API keys часто використовуються для:
публічних API,
інтеграцій,
сервісних акаунтів.
Наприклад:
Google Maps API працює через API keys.
Але API key сам по собі не дуже безпечний, тому що якщо його вкрасти — можна використовувати API від чужого імені.
Далі — Basic Authentication.
Тут клієнт передає:
логін,
пароль.
Зазвичай у заголовку Authorization.
Але важливо:
Basic Auth майже завжди використовується лише разом із HTTPS, інакше пароль може бути перехоплений.
Сьогодні Basic Auth використовується рідше, переважно:
у тестових середовищах,
внутрішніх системах,
або простих сервісах.
Тепер — OAuth2.
OAuth2 — це сучасний стандарт авторизації.
Саме через OAuth2 працює:
“Увійти через Google”,
“Увійти через Facebook”,
авторизація в багатьох мобільних застосунках.
Головна ідея:
користувач не передає свій пароль сторонньому сервісу.
Наприклад:
ви логінитесь через Google,
Google підтверджує вашу особу
і видає спеціальний access token.
Сторонній сервіс отримує лише токен, а не ваш пароль.
OAuth2 має різні потоки або flows:
Authorization Code Flow,
Client Credentials,
Refresh Token Flow.
Для студентів зараз головне зрозуміти:
OAuth2 дозволяє безпечно делегувати доступ між системами.
Далі — JWT.
JWT означає JSON Web Token.
Це спеціальний токен, який містить інформацію про користувача.
Наприклад:
userId,
роль,
час життя токена.
JWT складається з трьох частин:
header,
payload,
signature.
Сервер підписує токен цифровим підписом, щоб його не можна було підробити.
JWT дуже популярний у:
REST API,
SPA frontend застосунках,
мобільних додатках.
Наприклад:
після логіну backend повертає JWT,
а frontend потім надсилає його у кожному запиті.
І остання тема — mTLS та додаткові захисти.
mTLS — mutual TLS.
Це коли не тільки клієнт перевіряє сервер,
а й сервер перевіряє клієнта через сертифікати.
Такий підхід часто використовується:
у банках,
enterprise системах,
внутрішніх мікросервісах.
Окрім цього API часто захищають через:
rate limiting,
IP whitelist,
API Gateway,
WAF,
шифрування,
rotation токенів.
Тому безпека API — це не одна технологія, а цілий набір механізмів захисту.”

Слайд 8 — Версіонування API
Скрипт викладача:
“Тепер поговоримо про дуже важливу проблему в API — зміни.
Уявіть:
ви створили API,
його почали використовувати:
мобільний додаток,
frontend,
партнери,
інші сервіси.
І через рік вам потрібно змінити структуру відповіді або перейменувати поле.
Що станеться?
Старі клієнти можуть просто перестати працювати 😄
Саме для цього існує версіонування API.
Версіонування — це спосіб змінювати API без поломки існуючих інтеграцій.
Основна причина версіонування:
підтримка сумісності між різними версіями клієнтів.
Наприклад:
старий мобільний застосунок ще використовує v1,
а новий frontend уже працює з v2.
Тепер розглянемо основні підходи до версіонування.
Перший і найпоширеніший — через URI.
Наприклад:
/api/v1/users
/api/v2/users
Це дуже простий і зрозумілий підхід.
Його плюс:
легко побачити версію,
легко тестувати,
просто документувати.
Саме тому URI versioning найчастіше використовується у REST API.
Другий підхід — через HTTP заголовки.
Наприклад:
API-Version: 2
Або:
Accept-Version: v2
Тут URL не змінюється, але сервер дивиться на заголовок і вирішує, яку версію повернути.
Такий підхід більш “чистий” архітектурно, але складніший у підтримці.
Третій підхід — через media types або content negotiation.
Наприклад:
Accept: application/vnd.company.v2+json
Це часто використовується у великих enterprise API.
Але для початківців цей варіант менш популярний через складність.
Тепер дуже важливе поняття — backward compatibility.
Це сумісність назад.
Ідея проста:
нові зміни не повинні ламати старих клієнтів.
Наприклад:
якщо додати нове поле в JSON — це зазвичай ок.
Але якщо:
видалити поле,
змінити його тип,
або перейменувати —
старі клієнти можуть зламатися.
Тому хороші API намагаються максимально довго підтримувати сумісність.
Ще один важливий термін — deprecation.
Deprecation означає:
“ця версія API застаріла і скоро буде видалена”.
Наприклад:
компанія може сказати:
API v1 підтримується до кінця року,
після цього залишиться лише v2.
Зазвичай про це повідомляють:
у документації,
через email,
через спеціальні HTTP заголовки.
І останнє — план переходу для клієнтів.
У великих системах не можна просто сказати:
‘Ми все змінили, тепер переробляйте свої застосунки’.
Тому компанії:
підтримують кілька версій одночасно,
дають час на міграцію,
створюють migration guides,
і поступово переводять клієнтів на нову версію.
Це дуже важливо для стабільності продукту.
У реальному світі підтримка старих API — це окрема велика задача для команд backend-розробки.”

Слайд 9 — Документація та специфікації
Скрипт викладача:
“Ми вже говорили, що API без документації використовувати дуже складно.
Уявіть, що ви отримали API без опису:
які є ендпоінти,
які параметри передавати,
як виглядає відповідь,
які помилки можуть бути.
Тоді інтеграція перетворюється на вгадування 😄
Тому документація — це одна з найважливіших частин API.
Почнемо з OpenAPI та Swagger.
OpenAPI — це стандарт опису HTTP API.
Він дозволяє формально описати:
ендпоінти,
параметри,
формати відповідей,
статус-коди,
авторизацію,
схеми даних.
На основі OpenAPI часто працює Swagger UI.
Swagger — це зручний веб-інтерфейс для документації API.
Ви, мабуть, бачили сторінки типу:
/swagger
/swagger-ui
/api-docs
Swagger дозволяє:
переглядати API,
тестувати запити прямо з браузера,
дивитися приклади відповідей.
Для frontend або QA це дуже зручно.
Наприклад:
можна натиснути кнопку Execute і одразу відправити тестовий запит.
Далі — GraphQL schema та Playground.
У GraphQL API структура описується через schema.
Schema визначає:
які типи існують,
які поля доступні,
які запити можна робити.
Наприклад:
User,
Order,
Product.
Для роботи з GraphQL часто використовують Playground або GraphiQL.
Це інтерактивне середовище, де:
можна писати запити,
отримувати підказки,
бачити структуру API,
тестувати відповіді.
Тобто документація там частково генерується автоматично зі schema.
Ще одна дуже важлива тема — автогенерація SDK.
SDK — це набір готових бібліотек для роботи з API.
Наприклад:
замість ручного написання HTTP-запитів,
розробник може використовувати готовий Java або TypeScript SDK.
OpenAPI дозволяє автоматично генерувати:
клієнтські бібліотеки,
моделі даних,
приклади коду.
Наприклад:
Java SDK,
Kotlin SDK,
Python SDK,
TypeScript SDK.
Це значно пришвидшує інтеграцію.
Також хороша документація містить:
приклади запитів,
приклади помилок,
приклади авторизації.
Тому що реальні приклади часто корисніші за теоретичний опис.
І остання важлива тема — жива документація та changelog.
API постійно змінюються:
додаються нові поля,
нові ендпоінти,
нові правила авторизації.
Тому документація повинна оновлюватися разом із кодом.
Це називають “живою документацією”.
У хороших командах документація генерується автоматично прямо з коду або schema.
А changelog — це список змін API.
Наприклад:
додано новий endpoint,
deprecated старе поле,
змінено формат відповіді.
Це допомагає командам:
відслідковувати зміни,
оновлювати інтеграції,
і уникати поломок.
У великих компаніях хороша документація API — це критично важлива частина продукту.”

Слайд 10 — Обробка помилок і статус-коди
Скрипт викладача:
“Тепер поговоримо про помилки в API.
У реальному житті API майже ніколи не працює ідеально весь час 😄
Можуть бути:
неправильні дані,
проблеми авторизації,
недоступна база даних,
перевантаження сервера,
або проблеми мережі.
Тому дуже важливо правильно обробляти помилки.
Почнемо зі статус-кодів HTTP.
HTTP статус-код — це перше, що сервер повідомляє клієнту після запиту.
Коди поділяються на категорії.
2xx — успішні відповіді.
Наприклад:
200 OK
201 Created
4xx — помилки клієнта.
Тобто проблема в самому запиті.
Наприклад:
400 Bad Request — неправильні дані,
401 Unauthorized — користувач не авторизований,
403 Forbidden — доступ заборонено,
404 Not Found — ресурс не знайдено.
5xx — помилки сервера.
Наприклад:
500 Internal Server Error
503 Service Unavailable
Дуже важливо використовувати статус-коди правильно.
Наприклад:
не можна на всі помилки повертати 200 OK, а всередині JSON писати “error”.
Тому що:
frontend,
monitoring,
API Gateway,
автоматичні клієнти
орієнтуються саме на HTTP статуси.
Далі — структуровані тіла помилок.
Окрім статус-коду, сервер зазвичай повертає JSON з деталями помилки.
Наприклад:
{
"error": "VALIDATION_ERROR",
"message": "Email is invalid"
}
Хороші API повертають:
код помилки,
зрозуміле повідомлення,
іноді список полів з помилками,
requestId або traceId.
Це допомагає:
frontend,
QA,
backend розробникам,
і support-командам.
Тепер важлива тема — retry та backoff.
Іноді помилка тимчасова.
Наприклад:
сервер перевантажений,
мережа нестабільна,
сервіс тимчасово недоступний.
У таких випадках клієнт може повторити запит.
Це називається retry.
Але якщо тисячі клієнтів одразу почнуть нескінченно повторювати запити — сервер впаде ще сильніше 😄
Тому використовується backoff.
Backoff — це поступове збільшення затримки між повторними спробами.
Наприклад:
перша спроба через 1 секунду,
друга через 2,
третя через 5.
Це допомагає стабілізувати систему.
І остання дуже важлива тема — логування та кореляція запитів.
У сучасних системах один запит може проходити через:
API Gateway,
frontend backend,
auth service,
payment service,
notification service.
І якщо десь виникає помилка — треба зрозуміти, де саме.
Тому кожному запиту часто присвоюється спеціальний correlation ID або trace ID.
Цей ID передається між усіма сервісами.
Тоді в логах можна знайти весь шлях одного конкретного запиту.
Наприклад:
користувач каже:
‘У мене не пройшла оплата’.
І команда може знайти весь ланцюжок подій через trace ID.
У великих системах без цього дебажити проблеми майже неможливо.”

Слайд 11 — Асинхронні інтеграції
Скрипт викладача:
“До цього ми переважно говорили про класичні API-запити:
клієнт надсилає request,
сервер одразу повертає response.
Це називається синхронна взаємодія.
Але в реальних системах це не завжди зручно або можливо.
Наприклад:
платіж може оброблятися кілька секунд,
email може надсилатися у фоні,
аналітика може рахуватися хвилинами,
або сервіс тимчасово недоступний.
Тому дуже часто використовуються асинхронні інтеграції.
Асинхронність означає:
ми не чекаємо миттєвої відповіді.
Система може:
прийняти задачу,
поставити її в обробку,
і завершити пізніше.
Почнемо з webhook.
Webhook — це механізм “зворотного виклику”.
Ідея дуже проста:
замість постійних запитів:
‘Ну що? Дані вже готові?’ 😄
Сервіс сам повідомляє нас про подію.
Наприклад:
платіжна система після успішної оплати сама надсилає POST-запит у наш backend.
Тобто:
ми не опитуємо сервер кожну секунду,
а отримуємо повідомлення автоматично.
Webhook дуже популярні у:
Stripe,
GitHub,
Telegram,
CRM системах.
Наприклад:
GitHub може надсилати webhook після:
push у репозиторій,
створення PR,
запуску CI/CD.
Далі — черги повідомлень та брокери.
У великих системах сервіси часто не спілкуються напряму.
Між ними стоїть message broker — посередник для передачі повідомлень.
Найпопулярніші приклади:
RabbitMQ
Apache Kafka
RabbitMQ добре підходить для:
черг задач,
background processing,
класичних message queues.
Kafka більше орієнтована на:
великі потоки подій,
аналітику,
high-load системи,
streaming.
Наприклад:
в інтернет-магазині після створення замовлення можуть окремо запускатися:
оплата,
email,
аналітика,
доставка,
повідомлення користувачу.
І все це — через події та черги.
Тепер — Pub/Sub.
Pub/Sub означає Publish/Subscribe.
Один сервіс публікує подію,
а інші сервіси підписуються на неї.
Наприклад:
Order Service створив подію:
‘OrderCreated’.
Інші сервіси отримують її:
Notification Service,
Analytics Service,
Warehouse Service.
Це основа event-driven architecture.
У такій архітектурі системи реагують не на прямі запити, а на події.
Це дуже популярний підхід у:
мікросервісах,
cloud-системах,
великих distributed systems.
Переваги асинхронності:
краща масштабованість,
менше блокувань,
стійкість до навантаження,
можливість обробки у фоні.
Наприклад:
користувач натискає “Оформити замовлення”,
а система не чекає завершення всіх процесів прямо зараз.
Але є і складності.
Асинхронні системи:
складніше дебажити,
важче тестувати,
важче гарантувати порядок подій,
можуть виникати дублікати повідомлень.
Тому event-driven архітектура дає велику гнучкість, але вимагає більш складної інженерії.”

Слайд 12 — API Gateway та проміжне ПЗ
Скрипт викладача:
“Уявіть сучасну систему з десятками мікросервісів:
auth service,
payment service,
notification service,
user service,
analytics service.
Якщо frontend буде напряму звертатися до кожного сервісу — система стане дуже складною.
Тому в сучасних архітектурах часто використовується API Gateway.
API Gateway — це єдина точка входу в систему.
Frontend або мобільний застосунок звертається не до всіх сервісів окремо, а лише до gateway.
А gateway уже:
перенаправляє запити,
перевіряє авторизацію,
контролює навантаження,
і взаємодіє з внутрішніми сервісами.
Тобто gateway працює як “розумний посередник”.
Перша важлива функція gateway — routing.
Routing означає:
gateway вирішує, куди саме передати запит.
Наприклад:
/users → User Service,
/payments → Payment Service,
/orders → Order Service.
Frontend навіть може не знати, скільки сервісів є всередині системи.
Друга важлива функція — authentication та authorization.
Gateway часто перевіряє:
JWT токени,
API keys,
OAuth авторизацію.
Тобто безпека централізується в одному місці.
Третя функція — rate limiting.
Rate limiting обмежує кількість запитів від клієнта.
Наприклад:
не більше 1000 запитів на хвилину,
або 10 логінів за хвилину.
Це допомагає:
захищати API,
уникати перевантаження,
боротися з DDoS або brute-force атаками.
Ще одна важлива можливість — трансформація запитів.
Gateway може:
змінювати заголовки,
перетворювати формати,
додавати metadata,
або адаптувати старі API до нових сервісів.
Наприклад:
frontend надсилає один запит,
а gateway усередині викликає кілька мікросервісів та об’єднує відповідь.
Це називається aggregation.
Наприклад:
екран профілю користувача може одночасно потребувати:
дані користувача,
замовлення,
повідомлення,
статистику.
Gateway може зібрати все це в одну відповідь.
Переваги API Gateway:
централізація логіки,
простіший frontend,
єдина точка безпеки,
моніторинг,
логування,
кешування.
Але є і мінуси.
Gateway стає дуже важливим компонентом системи.
Якщо gateway впаде — може перестати працювати весь API.
Тому його потрібно:
масштабувати,
моніторити,
і правильно проєктувати.
Тепер кілька популярних прикладів.
Kong — популярний open-source API Gateway.
Він підтримує:
auth,
rate limiting,
plugins,
monitoring.
І ще дуже популярний варіант — AWS API Gateway.
Він активно використовується у cloud-архітектурах на AWS.
Також існують:
NGINX Gateway,
Apigee,
Traefik,
Istio для service mesh архітектур.
У сучасних distributed systems API Gateway майже став стандартом.”

Слайд 13 — Кращі практики та чекліст
Скрипт викладача:
“На завершення лекції давайте зберемо основні best practices для роботи з API та інтеграціями.
Тому що хороший API — це не просто “воно працює”.
Хороший API має бути:
зрозумілим,
стабільним,
безпечним,
масштабованим,
і зручним для інших команд.
Перше правило — прозорі контракти та повна документація.
API має чітко описувати:
які є ендпоінти,
які параметри потрібні,
які формати даних використовуються,
які можливі помилки.
Документація повинна бути:
актуальною,
зрозумілою,
і бажано інтерактивною.
Тому Swagger/OpenAPI сьогодні стали фактичним стандартом.
Друге правило — безпека, тестування та моніторинг “з народження”.
Це дуже важливий момент.
Безпеку не можна “додати потім”.
API від самого початку повинно враховувати:
авторизацію,
rate limiting,
HTTPS,
захист токенів,
валідацію даних.
Те саме з тестуванням.
API тестують:
unit тестами,
integration тестами,
contract testing,
performance testing,
security testing.
І дуже важливий моніторинг.
У production потрібно бачити:
кількість помилок,
latency,
навантаження,
статуси сервісів.
Для цього використовують:
logging,
tracing,
metrics,
dashboards.
Наприклад:
Kibana,
Grafana,
Prometheus.
Третє правило — план версіонування та сумісності.
API майже завжди буде змінюватися.
Тому потрібно заздалегідь продумати:
як додавати нові функції,
як підтримувати старі клієнти,
як повідомляти про deprecated API.
Різкі breaking changes — одна з найчастіших проблем інтеграцій.
Хороші команди:
підтримують backward compatibility,
роблять migration guides,
і дають час на перехід.
І останній важливий пункт — управління ключами, SLA та командна відповідальність.
API keys, JWT та секрети потрібно:
правильно зберігати,
регулярно оновлювати,
не хардкодити в коді.
Також у великих компаніях існує SLA — Service Level Agreement.
Це домовленість про:
uptime,
швидкість відповіді,
доступність API,
підтримку.
Наприклад:
API має працювати 99.9% часу.
І ще дуже важливо:
API — це відповідальність не лише backend-команди.
У хорошій інженерії:
frontend,
backend,
QA,
DevOps,
security engineers
працюють разом.
Тому що API — це основа взаємодії всієї системи.
І саме від якості API дуже часто залежить стабільність усього продукту.”
